# Migrating to Token Savior v3.0

Short version: **v3.0 flips the default profile from `full` (94 tools) to `lean` (58 tools).**

v2.8 keeps the old default but prints a one-line deprecation warning at server boot so the change does not surprise anyone when v3.0 lands.

## What changes

| | v2.8 (current) | v3.0 (breaking) |
|---|---|---|
| Default `TOKEN_SAVIOR_PROFILE` | `full` | `lean` |
| Tools advertised at handshake | 94 (14 159 tokens) | 58 (9 907 tokens) |
| Memory engine tools visible | yes | **no** (opt-in) |
| Reasoning / corpus tools visible | yes | **no** (opt-in) |
| Niche analysis tools (duplicate-classes, call-predictions, pack-context) | yes | **no** (opt-in) |

Tools never removed, only filtered from the manifest. Every dropped tool can be called by explicit name, or re-advertised by bumping the profile env var.

## Why

The handshake manifest was measured at 14 159 tokens in `AUDIT.md` Phase 0, which is above Claude Code's 10k auto-defer threshold. `lean` at 9 907 tokens slots just below the threshold, keeps the full code-navigation surface, and drops machinery that sees very little actual use outside of explicit memory workflows.

Anthropic's own research (cited in AUDIT.md Phase 3.6) shows tool-selection accuracy degrades past 30–50 tools. 58 is already high; 94 is too many.

## Three ways to keep the v2.8 behaviour on v3.0+

1. **Set the env var explicitly** on the server side:
   ```bash
   export TOKEN_SAVIOR_PROFILE=full
   ```

2. **Tell your MCP client to keep everything hot** — on Anthropic API, use `mcp_toolset` with `default_config.defer_loading: false`. See `docs/mcp_toolset.example.json` for a recommended layout.

3. **Pin to the v2.x line** in your dependency manifest:
   ```toml
   token-savior-recall = ">=2.7,<3"
   ```

## If you relied on memory tools visible by default

Everything `memory_*` is still exposed under `TOKEN_SAVIOR_PROFILE=full`. On `lean` they are still callable by exact name (the handlers stay wired), they just do not appear in `list_tools`. The auto-injection hooks (`hooks/memory-*.sh`) continue to operate because they talk to the database directly, not through MCP tools.

## Rollback plan

If v3.0 breaks your workflow, dropping back to v2.8 is `pip install token-savior-recall==2.8.*`. No schema changes, no data migration, your `.token-savior-cache.json` carries over.

## What v3.0 does *not* change

- Tool names or signatures.
- Memory schema.
- Hook lifecycle.
- Cache format.
- Python version support (>= 3.11).
- Nomic embedding model.
- Default behavior when the user sets `TOKEN_SAVIOR_PROFILE` explicitly.

## Timeline

- **v2.8.0** (target: 2026-04-24) — emits the deprecation warning, ships the migration guide, `mcp_toolset.example.json`, docs reconciliation, A2 list caps, B3 file watcher.
- **v2.9.0** (target: 2026-05) — may include A5 telemetry, threshold tuning based on collected data, `_ULTRA_INCLUDES` adjustments.
- **v3.0.0** — flips the default. No earlier than 2 weeks after v2.8.0 so users have time to notice the warning and migrate.
